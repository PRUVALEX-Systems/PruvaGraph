# ADR-001: Single SQLite Database for All Five Modules

**Status:** Accepted  
**Date:** 2025-01  
**Authors:** PRUVALEX Engineering  
**Deciders:** Kamal (Founder)

---

## Context

PRUVALEX is a VS Code extension containing five modules: DriftGuard, ContextLens, GhostMemory, RulesForge, and TaskWeaver. Each module requires persistent storage for its data (indexed symbols, token logs, memories, rules, checkpoints respectively).

The most natural first instinct for a multi-module architecture is to give each module its own database file. This document explains why that instinct is wrong for this specific system.

---

## Decision

**All five modules share one SQLite database file: `pruvagraph.db`**, located in VS Code's `globalStoragePath`. Each module is granted access only through a scoped `ModuleRepository` instance, namespaced by the prefix `<moduleid>_`.

---

## Options Considered

### Option A: One SQLite DB per module (5 files)
Each module owns `driftguard.db`, `ghostmemory.db`, etc.

**Pros:**
- Maximum isolation — a corrupt driftguard table cannot affect ghostmemory data.
- Simpler per-module schema ownership.

**Cons:**
- Five concurrent `better-sqlite3.Database` connections from one process. SQLite is designed for one writer. Concurrent writers across five files is not a correctness problem but *is* a performance and resource problem in a VS Code extension process.
- ContextLens cannot join `token_ledger` with module-specific data for aggregate cost reporting without cross-DB queries (which SQLite does not support natively).
- Five DB files in `globalStoragePath` is confusing for users and enterprise IT administrators.
- Each module must independently manage its migration lifecycle — five separate migration runners with no coordination.

### Option B: One SQLite DB, all tables in one schema (no namespacing)
Single file, all tables flat (`entries`, `rules`, `checkpoints`, etc.).

**Pros:**
- Simplest schema.

**Cons:**
- No audit trail for which module owns which table. As the extension grows, table ownership becomes opaque.
- Name collisions are guaranteed as modules add tables (`entries` claimed by both GhostMemory and a future module).
- Impossible to enforce per-module data boundaries (e.g., "disable GhostMemory and delete its data only").

### Option C (CHOSEN): One SQLite DB, namespaced tables per module
Single `pruvagraph.db` file. Table naming convention: `<moduleid>_<tablename>`.

**Pros:**
- One DB connection (`better-sqlite3.Database` is synchronous and single-connection by design — this is its ideal usage pattern).
- ContextLens can query `token_ledger` with zero overhead — same DB, same connection.
- Schema is self-documenting: `SELECT name FROM sqlite_master WHERE type='table'` immediately reveals which module owns what.
- One migration runner in `core-engine` coordinates all five modules' migrations in order, preventing partial-upgrade states.
- Disabling a module cleanly means "do not run queries against `ghostmemory_*` tables" — no data loss, no file deletion needed.
- Auditable by enterprise IT: one file, `sqlite3 pruvagraph.db ".dump"` reveals everything.

**Cons:**
- A schema bug in one module's migration can block all other modules from activating (mitigated by migration transaction isolation — each migration runs in its own `BEGIN TRANSACTION`).
- Module data is co-located — a user cannot selectively back up just GhostMemory data. (Mitigated by the `scope()` API which enables per-module export functions.)

---

## Consequences

- The `IGraphStore.scope(modulePrefix: string)` API is the enforcement mechanism. Modules may only query tables that start with their assigned prefix. This is enforced at the TypeScript interface level and audited by the forbidden-import lint rule.
- The migration runner in `core-engine/src/db/connection.ts` is the single source of truth for schema versioning. No module ships its own migration runner.
- `pruvagraph.db` must be excluded from `.gitignore` for users (it is a runtime artifact, not a source file).
- Max DB size is contractually capped at 50MB via auto-vacuum + TTL on `token_ledger`. This is enforced by the `driftguard.bench.ts` benchmark.
- Enterprise deployments requiring encrypted at-rest storage must use filesystem-level encryption (e.g., BitLocker, FileVault) rather than SQLite's SEE extension, to avoid licensing complexity.
