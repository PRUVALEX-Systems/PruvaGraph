# ADR-003: Auto-Venv Python Engine Bootstrap (PruvaGraphRunner)

**Status:** Accepted  
**Date:** 2025-01  
**Authors:** PRUVALEX Engineering  
**Deciders:** Kamal (Founder)

---

## Context

PRUVALEX requires the PruvaGraph Python engine to generate `graph.json` — a dependency graph that backs DriftGuard's indexer and RulesForge's AST-layer detector. The Python engine must be invoked from within a TypeScript VS Code extension process.

Three deployment options were evaluated:

1. **PyInstaller** — ship a pre-compiled binary per OS architecture.
2. **Docker sidecar** — run PruvaGraph as a containerized process.
3. **Auto-Venv bootstrap** — ship Python source, bootstrap a virtual environment at runtime.

---

## Decision

**Use Auto-Venv (virtual environment bootstrapping)** via `PruvaGraphRunner` in `core-engine/src/runner.ts`.

---

## Options Analysis

### Option A: PyInstaller Binaries

**What it does:** Compiles the Python engine + all dependencies into a single OS-specific binary. Ships three binaries: `pruvagraph-win32.exe`, `pruvagraph-darwin-arm64`, `pruvagraph-linux-x64`.

**Pros:**
- Zero runtime dependency on Python.
- Fastest startup (no pip install).

**Cons:**
- **VSIX size explodes to > 200MB** (three binaries, each 50–100MB). VS Code Marketplace has a recommended limit of 50MB and a hard limit of 1GB — but Marketplace ranking algorithms actively penalize large extensions. The target is < 10MB.
- **Enterprise/EDR rejection.** Unsigned `.exe` binaries from an unknown publisher will trigger Windows Defender and enterprise EDR systems (CrowdStrike, SentinelOne) in banking environments. Getting code-signed certificates for three platforms adds weeks and ongoing cost.
- **CI/CD complexity.** Requires three separate build agents (Windows, macOS Intel, macOS ARM). Current team size: 1. This is not proportionate.
- **Update cycle.** Every Python dependency update requires rebuilding and re-shipping all three binaries.

### Option B: Docker Sidecar

**What it does:** Runs PruvaGraph in a Docker container, communicates via stdio or HTTP.

**Pros:**
- Maximum isolation.
- Reproducible environment.

**Cons:**
- **Requires Docker Desktop** to be installed and running. Enterprise developers often do not have Docker Desktop (licensing cost, IT approval). This is a non-starter for the Frankfurt banking segment.
- **Startup latency.** Container cold start adds 5–15 seconds to the first graph generation. Unacceptable for an extension activation flow.
- **Port conflicts.** HTTP communication requires a local port — enterprise firewalls and host-based IDS systems may flag or block this.

### Option C (CHOSEN): Auto-Venv Bootstrap

**What it does:** Ships Python source code (`pruvagraph/`) inside the VSIX. At activation, `PruvaGraphRunner`:
1. Locates `python3` or `python` on `$PATH`.
2. If `.venv_pruvagraph` does not exist in `globalStoragePath`: creates it, runs `pip install -e ./pruvagraph`.
3. Runs `python -m pruvagraph.cli <workspaceRoot>` as a detached child process.
4. Pipes stdout/stderr to a VS Code Output Channel (`PRUVALEX — PruvaGraph Engine`).

**Pros:**
- **VSIX stays under 10MB** (Python source is small; only runtime data is generated).
- **Enterprise-safe.** Python source code is auditable. No unsigned binaries. No network calls during install (pip uses the bundled `pruvagraph/` package — it is not fetched from PyPI).
- **Cross-platform with zero CI complexity.** One build produces a VSIX that works on all platforms where Python is installed.
- **Graceful degradation.** If Python is absent, the extension notifies the user once and continues operating with TypeScript-only AST analysis. All five modules activate; DriftGuard and RulesForge lose the graph-enhanced features only.
- **Update simplicity.** Python engine updates are just source file changes — no binary recompilation.

**Cons:**
- **Requires Python ≥ 3.9** to be installed. This is non-negotiable. However: all target enterprise environments (banking dev workstations) have Python — it is a prerequisite for any ML tooling.
- **First-run pip install latency** (10–30 seconds if internet-dependent). Mitigated by: (a) using the bundled package (no network required), (b) showing a native VS Code progress notification so the user is not surprised.
- **.venv_pruvagraph disk footprint** (~50–80MB for numpy + tree-sitter Python bindings). Mitigated by placement in `globalStoragePath` (not workspace) and documented cleanup command.

## Graceful Degradation Matrix (Non-Negotiable)

```
State                              Behavior
─────────────────────────────────────────────────────────────────────
Python not found                  → One-time notification + degrade
Python found, pip fails           → Log error to Output Channel + degrade
Venv created, import fails        → Log error + degrade
Graph generated, stale (>24h)     → Re-run silently in background
Graph generated, current          → Full feature mode
User triggers manual rebuild      → pruvagraph.initializeGraph command
```

"Degrade" means: DriftGuard operates with TypeScript-only type index (no cross-file dependency graph). All other modules (ContextLens, GhostMemory, RulesForge rules-only mode, TaskWeaver) are unaffected.

The extension host process must never crash due to `PruvaGraphRunner` failure. Every code path in `runner.ts` is wrapped in try/catch with explicit user notification.

---

## Security Considerations

- The bundled `pruvagraph/` Python package must be pinned to exact dependency versions (`requirements.txt` with hashes).
- `pip install` must use `--no-index --find-links ./pruvagraph/dist/` to install from the bundled wheel, not from PyPI. This eliminates supply-chain risk during installation.
- The `.venv_pruvagraph` directory must be excluded from VS Code's file watcher to prevent excessive `onFileChanged` events.
- The child process spawned by `PruvaGraphRunner` must have restricted permissions (no `shell: true`, explicit args array, no `env` passthrough of secrets).

---

## Consequences

- `PruvaGraphRunner` lives in `core-engine` (not a module) because its lifecycle is independent of any module's activation state.
- The `pruvagraph.initializeGraph` command is registered in `extension.ts`, not in any module.
- `.venv_pruvagraph/` is listed in `.vscodeignore`, `.gitignore`, and the troubleshooting guide.
- Enterprise air-gap deployments must pre-stage the `.venv_pruvagraph` on developer workstations via IT provisioning scripts. The compliance brief (see `docs/enterprise-compliance-brief.md`) documents this procedure.
