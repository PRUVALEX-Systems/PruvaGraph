# ADR-002: Hexagonal Architecture — Ports & Adapters for All Core Services

**Status:** Accepted  
**Date:** 2025-01  
**Authors:** PRUVALEX Engineering  
**Deciders:** Kamal (Founder)

---

## Context

PRUVALEX has five modules that all need access to: a database, an MCP client, workspace file information, and an event bus. The naive implementation is for each module to directly instantiate what it needs:

```typescript
// The naive (forbidden) approach
import Database from "better-sqlite3";
import { Client } from "@modelcontextprotocol/sdk/client/index.js";

export class GhostMemoryModule {
  private db = new Database("ghostmemory.db");    // ❌ Own DB instance
  private mcp = new Client(...);                   // ❌ Own MCP client
}
```

This naive approach is the "five separate extensions" failure mode in disguise. It looks like a unified extension but behaves like five isolated systems that happen to share a package.json.

---

## Decision

**All core services (database, MCP transport, workspace context, event bus) are accessed exclusively through typed interfaces (ports).** Concrete implementations live only in `core-engine`. Modules receive ports through constructor injection.

The four mandatory ports are:

```typescript
// packages/core-engine/src/ports/

IGraphStore      // Wraps the one SQLite connection
IMCPTransport    // Proxies the one MCP client
IWorkspaceContext // Wraps file tree, package.json, AST cache
IEventBus        // In-process pub/sub
```

---

## Why Hexagonal (Not Just Dependency Injection)

Hexagonal (Ports & Adapters) goes further than simple DI. The key distinction:

| Plain DI | Hexagonal Ports & Adapters |
|---|---|
| Injects concrete class | Injects an interface (port) |
| Module knows what it's getting | Module has no knowledge of what implements the port |
| Swap requires recompile | Swap requires zero module code changes |
| Testing requires mocks of real classes | Testing uses simple stub implementations of the port |

For PRUVALEX specifically, the hexagonal constraint means:
- **GhostMemory has zero knowledge that it is talking to SQLite.** It calls `this.store.scope("ghostmemory").get(id)`. The backing store could be SQLite, in-memory, or a remote DB — GhostMemory's code does not change.
- **DriftGuard has zero knowledge it is talking to an MCP SDK client.** It calls `this.mcp.call("pruvagraph", "validate_symbol", args)`. The routing, token counting, and retry logic are invisible to DriftGuard.
- **All five modules can be unit-tested with three-line stubs** that implement the port interfaces. No test requires a real SQLite file or an MCP server.

## Enforcement Mechanism

Architectural decisions that rely on developer discipline fail. This one relies on tooling:

**1. TypeScript interfaces as the physical contract:**
```typescript
// module-ghostmemory/src/index.ts
export class GhostMemoryModule implements PRUVALEXModule {
  constructor(
    private store: IGraphStore,       // port — not GraphStore
    private mcp: IMCPTransport,       // port — not MCPRouter
    private workspace: IWorkspaceContext,
    private events: IEventBus<PRUVALEXEventMap>
  ) {}
}
```
TypeScript's type system enforces that the module can only call methods defined on the port interface. Methods on the concrete class that aren't in the interface are invisible.

**2. ESLint forbidden-import rule (runs in CI and in editor):**
```javascript
"no-restricted-imports": ["error", {
  "paths": [
    { "name": "better-sqlite3", "message": "Use IGraphStore port." },
    { "name": "@modelcontextprotocol/sdk", "message": "Use IMCPTransport port." }
  ]
}]
```
This turns an architectural convention into a build error. A developer cannot accidentally import `better-sqlite3` in a module file — it fails the linter before the PR is even opened.

**3. Grep assertion in CI:**
The `ci.yml` workflow explicitly greps for forbidden imports in `packages/module-*/src/` and fails the build if any are found. This provides a second line of defense that runs even if the ESLint config is misconfigured.

## Consequences

**Positive:**
- Any port's concrete implementation can be swapped without touching module code. Example: replacing the local embedding model in GhostMemory's store implementation has zero impact on GhostMemoryModule itself.
- Unit tests for all five modules are fast (no I/O, no network) and require no test infrastructure setup.
- New contributors can implement a module by only reading `packages/shared-types/src/module-contract.ts` — they never need to understand the internals of `core-engine`.
- The architecture is auditable: `grep -r "new Database" src/` produces exactly one result in the entire codebase.

**Negative:**
- More boilerplate at the composition root (`di/container.ts`) — each interface implementation must be explicitly wired.
- Port interfaces must be versioned carefully. Adding a method to `IGraphStore` requires updating all stub implementations in tests.
- Port design is a skill. Poorly designed ports leak implementation details (e.g., an `IGraphStore` method that returns a `Statement` object couples the port to SQLite). Port methods must deal only in plain data types.
